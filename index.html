Project Ideas

Overview

The project that comes up in my mind is a (mobile) game involving simple physics. The objective of this game is to simply have a ball (or any other object) to not touch the ground of the stage as long as possible with given tools (platforms, etc.). There are some other thoughts that come in mind whilst I developed the ideas, such as other game modes/types or online aspect implementations.

Motivation

This idea comes up in my mind whenever I am outside and does not have anything to do, more particularly while waiting for something. Playing games on mobile phone seems to be the practical solution here, except when said mobile phone have limited amount of battery power and power socket might not be found. Moreover, some people cannot afford to buy or have a mobile phone that have the necessary specifications to run a certain game. While there are already many games that have taken these problems into account, none of them (as far as my research concerned) play exactly the same as my project idea. The game that is also simple compared to the other mobile games in the market, therefore only relatively little skill/experience needed to develop the game or the prototype of it.

Description

Generally, this game has an objective of dropping a ball (or other kind of object applicable) until it reaches the ground of the stage within the time limit. This limit, however, is not an upper but instead lower limit (minimum not maximum). Therefore, players are given several “tools” that is in-game objects that would help them maintain the ball from falling to the ground before the time limit reached. These objects can be general objects, such as platforms, to the more complicated ones, such as gears. Note that the player could easily keep the ball from falling at all with even one platform, but this won’t complete the game’s objective which explicitly stated to have the ball reach the ground. As such, players are expected to slow down the fall of the ball whilst avoiding the ball itself to stuck. For this, the players would need to mind the physics around the stage and use them to their advantage. The stage itself could be modified for increased challenge in further stages. These include, but not limited to, limitation of tools, time limit, gravity, density of the ball and layout of the stage itself.

The game does not need to end only with specified features. For instance, the game mode can be changed from reaching a certain time limit to distance or even height. This will, in turn, changes the game almost completely as the objective is now different. Naturally, this will also make the player to employ different strategies or approach. New items/tools could also be provided to the player to give convenience or to help them solve the stage as some objective might not be able to be completed using the original tools. Moreover, online features could be integrated in the game, which can be implemented in some ways. One of the most widely used features is high score. With this, players can post their best score, which form varied on the objective of the stage. This high score can be posted publicly to the “leaderboard” which can be seen publicly. The user can also choose to not publicly upload their score and instead save it locally in their device (which is usually the common protocol when the user does not upload their data). Additionally, a feature which lets players to make their own stage, which have been done some games such as Super Mario Maker, can be added. This will give the players an opportunity to share their creation across the platform and let others play and review it or privately uploaded to be able to be seen only by friends. Needless to say, this might be wrongly/irresponsibly used by some. As such, some anticipation would be needed should the feature be added, such as user’s ability to report certain stages uploaded online or manually check the database. Finally, it is important to note that while all these features are supposedly complementary, they will increase the game’s “replayability” by much, which is very crucial in the game industry.

Tools and Technologies

As stated before, this game is relatively simple compared to the general mobile games in the market. As such, tools and technologies utilised can be something with relatively low and simple specifications. Some fundamental tools and technologies that can be used are as follow:

  • Program for making the game

    • The tool for this particular game does not necessarily restricted to have the ability to handle large projects. The program that might be used are Unity, Construct2 or similar.

  • Laptops or PC (Personal Computers)

    • In addition to the software for creating the game, a hardware is also in order. Since the game requires little specifications, any laptop that support windows and can run emulators (should the testing does not use a mobile phone) would suffice. It is worth noting that the better the specifications are, the faster the game development will generally be.

  • Smartphone or emulator

    • A smartphone, preferably android, would be needed to test the product/prototype. Alternatively, an emulator could be used from laptops/PC instead of smartphone.

Skills Required

  • Understanding of programming language (language differs depending on the tool used) and skills (logical thinking, etc.)
  • Basic designing skills in designing the in-game object

Outcome

With the low specifications for the game, this game would supposedly be a decent “time-killer” whilst consuming as little battery power as possible and provide accessibility to those who does not have mobile phones with high specifications. Should the product come out successfully, additional features can be added with users’ feedback in mind. As for the impact to the development team, more knowledge of programming language can be acquired, along with some other skills, such as designing and communication skills.
